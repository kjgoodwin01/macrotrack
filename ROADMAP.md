# MacroTrack — App Store & Revenue Roadmap

Goal: downloadable app store app generating subscription revenue, beating MFP and MacroFactor.

---

## Phase 1 — Foundation

- [x] Supabase Auth — email magic link sign-in/sign-up
- [x] Auth gate in onboarding — required before any data entry, no anonymous mode
- [x] AccountCard in settings — shows signed-in email, sign-out
- [x] auth_migrate worker route — moves anonymous Supabase rows to real auth UID on first sign-in
- [x] Apple Sign-In + Google Sign-In — required by App Store guidelines for apps with any login. Google currently opens Safari (needs @capacitor/browser + deep link handling). Apple sign-in doesn't work yet (needs native plugin).
- [x] RevenueCat + Apple In-App Purchase — paywall on AI coach + meal plans via RevenueCat SDK. iOS App Store only; Android/Google Play Billing deferred.

---

## Phase 2 — Competitive Parity

- [x] Adaptive TDEE algorithm — EMA trend weight + reverse-engineered TDEE from real calorie/weight data; falls back to 7-day heuristic while data accumulates; trend line on chart
- [ ] Apple Health / Google Fit sync — required for serious fitness app users
- [ ] Recipe builder — log multi-ingredient meals
- [x] Capacitor wrapper — wraps existing web code as a native iOS + Android app for app store submission. App runs on device. Email auth working. Nav safe area correct. Install banner hidden in native. OAuth (Google/Apple) still needs fixing before submission.
- [ ] Vite + React build migration — replace single 8k-line HTML file with proper build system before it becomes unmanageable

## Phase 3 — Differentiation

- [x] Native iOS push notifications — APNs via @capacitor/push-notifications. WKWebView has no PushManager so web push was silently failing ("Push not supported"). Fixed by routing Capacitor context through APNs; worker generates APNs JWT (ES256) and sends via api.push.apple.com. Scheduled crons (daily reminder 12 PM EDT, protein gap 5 PM EDT, weekly report Sunday 9 AM EDT) fire from Cloudflare Worker. Four bugs fixed 2026-04-25 before notifications actually delivered: (1) scheduled handler was using anon-key sb() to query Supabase — protected tables returned nothing, so notifications were silently skipped; fixed by setting SUPABASE_SERVICE_ROLE_KEY as a Cloudflare secret so sbAdmin bypasses RLS. (2) notify_reminder defaulted to false so noon reminders never fired. (3) push_subscriptions upsert was missing ?on_conflict=device_id, causing duplicate key errors on re-subscribe and silently failing. (4) weekly report cron was set to 14 UTC (10 AM EDT) instead of 13 UTC (9 AM EDT). All fixed in builds 6–8 + worker redeploy. Additional fixes 2026-04-26: (5) APNS_PRIVATE_KEY was the ASC API key, not a dedicated APNs Auth Key — all pushes were returning InvalidProviderToken (403) and silently failing without deleting the subscription. Fixed by creating a new APNs Auth Key (X8RVJTKXKV, Production & Sandbox) and updating Cloudflare secrets. (6) Development/Xcode builds use sandbox APNs tokens which are rejected by the production endpoint (api.push.apple.com) with BadDeviceToken (400). Fixed by adding automatic sandbox endpoint fallback in sendApnsPush — tries production first, retries api.sandbox.push.apple.com on BadDeviceToken. TestFlight/App Store builds hit production directly; dev builds fall back to sandbox transparently. (7) Subscription deletion was too aggressive — any 400 from APNs deleted the subscription row, including JWT errors and payload errors. Fixed to only delete on 410 (Unregistered) or 400 with explicit BadDeviceToken in response body. (8) Added admin_test_apns worker endpoint for on-demand push testing by device_id without waiting for a cron. (9) TestFlight users saw "Push not supported" because build 6 predated the native push code in app.html — fixed in build 9.
- [x] Food search overhaul (build 10, 2026-04-27) — multiple compounding failures fixed: (1) Cloudflare Worker fetch handler was missing `ctx` parameter so all KV cache writes were silently dropped since the runtime cancelled dangling promises on response return; fixed by adding `ctx` as third parameter and replacing fire-and-forget puts with `ctx.waitUntil()`. (2) AI orchestrator `max_tokens` was 800, truncating JSON output mid-string and causing all AI calls to return null; raised to 2000. (3) Sequential DB→AI pipeline (5–8s total) replaced with stale-while-revalidate: DB results returned immediately (~1.2s), AI enrichment runs in background after response is sent and overwrites cache on next search (~200ms). (4) Two-level KV cache: word-sorted normalized key collapses word-order variants to the same entry; flavor-stripped base key collapses "kirkland peach energy drink" / "kirkland tropical energy drink" to the same base entry. (5) Brand scoring fix: generic words (energy, drink, co, foods) excluded from brand-match signal so "Monster Energy Company" no longer beats "Kirkland Signature" on a kirkland query. (6) AI macro accuracy fix: AI orchestrator now returns a `ci` (candidate index) field referencing the original DB entry; worker uses real USDA/OFF `foodNutrients` for those results instead of AI-regenerated values. (7) Community food database: new `report_food` worker endpoint accepts verified food data on every log; writes to `ureport:` KV keys with 5-year TTL; search handler reads `ureport:` and `search:` in parallel and always surfaces community-verified entries first; AI-generated entries are not reported (only USDA/barcode data). One user's barcode scan benefits all users permanently.
- [x] Push notification accuracy + workout tab fixes (2026-04-27) — three bugs fixed: (1) False "haven't logged" noon notification: auth_migrate was migrating food_entries and weight_log to the new auth UID but leaving push_subscriptions under the old anonymous device_id; cron queried food entries under the stale subscription device_id, found none, and fired incorrectly; fixed by migrating push_subscriptions in auth_migrate (delete stale row if new device_id already has a subscription, otherwise update it). (2) Workout edit sheet SAVE button hidden behind tab bar on iOS native: edit-sheet padding-bottom only accounted for the home indicator safe area, not the nav bar height; fixed with html.native .edit-sheet { padding-bottom: calc(var(--nav-h) + 10px) }. (3) Workout edit sheet glitchy half-screen-to-full-screen animation: the shared sup keyframe uses translateX(-50%) which is correct for left:50% positioned elements but wrong for left:0/right:0/margin:auto — during animation the sheet was shifted 50% off-screen left, then snapped to center when animation ended with no fill-mode; fixed by giving .edit-sheet its own sheet-up keyframe using translateY only. (4) Deleted workouts repopulating on app reopen: two compounding bugs — sync_workout was POSTing to workout_sessions without ?on_conflict=device_id,log_date so every update after the first silently failed the unique constraint and the cloud always held the original workout state; and on load the merge logic unconditionally let cloud win per date so even a correct local deletion was overwritten; fixed by adding proper upsert and introducing mt_workouts_ts timestamps so the merge keeps whichever version (local or cloud) was written more recently.
- [x] UX + data quality fixes (2026-04-27) — four issues resolved: (1) Coach message history was unbounded and would eventually exhaust localStorage; capped at a rolling 50-message window. (2) Recents list in the add-food modal was computed once at mount from a localStorage snapshot; converted to a useMemo derived from live React entries state so it updates immediately when food is logged. (3) Workout session stat showed raw total poundage ("Volume lb", e.g. 3.4k) which is meaningless — replaced with Top Set (heaviest weight lifted that session, e.g. 225lb). (4) Custom meal editor required manually typing macros for each item with no way to look up a food; added a 🔍 search button per item that queries the food database inline and auto-fills name, serving, and all macro fields on selection.
- [x] Streak tracking (2026-04-28) — consecutive logged days computed from mt_entries, displayed as 🔥 N badge at the bottom-right of the CalHero card (amber at 7+ days, muted grey below); included in AI coach context so the coach can reference and react to streaks.
- [x] iOS home screen widget (2026-04-28) — WidgetKit extension (MacroWidgetExtension) embedded in the app bundle. Small size shows calories remaining + streak; medium size adds protein/carbs/fat progress bars. Data flows from the web layer via a WKScriptMessageHandler (syncWidgetData) that writes to a shared App Group UserDefaults container (group.live.macrotrack.app) and calls WidgetCenter.reloadAllTimelines() on every food log or goal change. Widget refreshes automatically every 30 minutes as a fallback. Tapping the widget deep-links into the app. iOS 17+ uses containerBackground(.clear); iOS 16 falls back gracefully.
- [x] Widget white outline fix (2026-04-28) — iOS 17+ containerBackground was set to .clear, causing the system's default white container shape to bleed around the dark widget content. Fixed by passing darkBg directly to containerBackground and removing the now-redundant ZStack background fills in SmallMacroView and MediumMacroView.
- [x] Dark mode app icon (2026-04-28) — Implemented iOS 18 dark/tinted home screen icon support. Root issue: the old multi-size asset catalog format (idiom: iphone with per-size entries) is silently ignored by actool in Xcode 26 — dark variants must use the new universal single-size format (idiom: universal, platform: ios, size: 1024x1024). Also required ASSETCATALOG_COMPILER_INCLUDE_ALL_APPICON_ASSETS = YES in both Debug and Release build configs. Dark icon uses a transparent background (iOS applies its system charcoal #1C1C1E automatically, matching every other app) with a white outline-only (hollow) left M and a solid vivid blue (#409CFF) right M + underline — a distinct dark-mode treatment that differentiates clearly from the light icon. Outline generated via circular disk morphological erosion for uniform stroke width.
- [x] Push notification false-fire fix (2026-04-28) — Daily reminder at noon was firing even when food had been logged. Root cause: push subscription device_id could become stale after re-authentication (original anonymous mt_ ID vs. Supabase auth UUID), so the cron queried food_entries under the wrong ID and found nothing. Fixed in two parts: (1) scheduled handler now resolves canonical device_id via email-based profile lookup before checking entries (applied to both daily reminder and protein gap checks); (2) push_subscribe_apns handler deletes stale subscription rows sharing the same APNs endpoint but a different device_id whenever a new subscription is registered, preventing orphan rows from accumulating.
- [x] Widget app name truncation fix (2026-04-28) — "MACROTRACK" label in the small widget was wrapping to two lines ("MACROTRAC / K") due to excessive letter-spacing (tracking 0.8). Fixed by reducing tracking to 0.3 and adding lineLimit(1).
- [x] Food log qty persistence fix (2026-04-28) — Two bugs: (1) serving label (e.g. "4 oz") never updated when qty multiplier was changed in the edit modal — fixed by stripping any existing Nx prefix and rebuilding it with the new qty on save. (2) qty was not stored on the entry so reopening the edit modal always showed qty=1 — fixed by persisting qty on the entry and scaling from per-unit base macros (item.p / item.qty) when qty is changed again.
- [x] NLP describe multi-select (2026-04-28) — Describe tab previously forced add-all-or-one-at-a-time. Replaced per-item "Add" buttons with tap-to-toggle checkboxes (all default checked); button label updates dynamically ("ADD ALL 6 ITEMS" vs "ADD 4 OF 6 ITEMS"); deselected items fade to 45% opacity. Users can now select any subset and log them in one tap.
- [x] Accurate exercise calorie burn (2026-04-28) — Three compounding bugs fixed: (1) Cardio exercises logged by distance only (e.g. "cycling 6.46 miles") produced zero calorie credit because calcExerciseCal required ex.duration — fixed by estimating duration from distance using activity-appropriate speeds (cycling 12 mph, running 6 mph, walking 3 mph, swimming 2 mph). (2) calcExerciseCal defaulted to 160 lbs if no weight was logged; now always uses the user's most recent weight log entry. (3) Age and sex from onboarding were used to compute TDEE but then discarded — never persisted to mt_goals — so calorie estimates were identical regardless of who was logging. Fixed by saving age, sex, heightCm, and activity into the goals object at onboarding finish and passing them through to calcExerciseCal. Sex applies a 10% reduction for female (body composition difference); age applies a 2% per-decade reduction after 20 (floored at 80%). GoalsModal onSave now preserves these fields so a macro edit doesn't wipe the profile data.
- [x] Profile section in Settings (2026-04-28) — New "Profile" row in Settings (between Account and Coach Personality) opens a full-screen sheet where users can view and edit sex, age, height, activity level, and current weight. Sheet fades in at full height (92dvh) to avoid the half-screen-then-full-screen animation glitch caused by height reflow during the slide-up animation. Includes a live TDEE preview that updates as fields change. Current weight is read-only, pulled from the most recent weight log entry with date shown; tapping "Log →" opens the weight modal. Onboarding data auto-populates the profile for new users. Existing users can fill it in once and all exercise calorie calculations immediately become personalized.
